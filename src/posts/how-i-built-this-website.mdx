## The Tech Stack Decision

The first real decision was choosing a tech stack. I narrowed it down to two options:

**Option 1: Astro** — simpler to learn, blazing fast, great for mostly-static sites.

**Option 2: Next.js** — the industry standard for React-based websites, more job-relevant, backed by Vercel.

I chose **Next.js 15** for one reason: I'm pursuing a web development degree and React is the skill that opens the most doors. Astro would have been easier, but Next.js makes me a better developer long term. Sometimes the harder path is the right one.

For hosting I chose **Vercel** — the company that builds Next.js. Deploying a Next.js site to Vercel is like plugging a lamp into the outlet it was designed for. One click, zero configuration, automatic SSL, and every GitHub push triggers a new deploy automatically.

**Time taken: about 30 minutes** from project initialization to first deploy.

## The Brand Identity

Before writing a single line of code, I knew the brand had to feel different from the typical local marketing agency — bold, transparent, and built for contractors rather than corporate clients.

### Color palette

I'm originally from North Carolina and I bleed Carolina and Duke blue. Those two colors became the foundation:

- **Carolina Blue `#7BAFD4`** — primary accent, used for headings, links, highlights, and CTAs
- **Duke Blue `#012169`** — secondary, used for borders, backgrounds, and depth
- **Background `#0a0a0a`** — near-black, not pure black, which feels softer on the eyes
- **Surface `#141414`** — card and navbar backgrounds, creating subtle depth layers

### Typography

After experimenting with Poppins early on, I landed on a combination that gives the site genuine personality:

- **Fraunces** — an optical serif with italic character for headings. It's editorial without being stuffy, and the italic style adds a human quality that sans-serif fonts can't match.
- **Urbanist** — a clean geometric sans-serif for body text. Highly legible, modern, and pairs beautifully with Fraunces without competing with it.

The pairing is like a well-dressed person — the serif is the personality, the sans-serif is the professionalism.

### The Logo

I built the logo entirely in code as a reusable SVG React component. The concept is a **location pin** — a filled Carolina Blue circle with a clean white stem and point, contained in a Duke Blue rounded square. It communicates "local" immediately and without ambiguity.

The design went through several iterations. I started with an abstract signal concept — concentric rings suggesting local reach — but it felt too generic. After experimenting in Canva, I landed on the location pin: simpler, bolder, and instantly tied to the idea of being found in a specific place.

Three key requirements drove every decision: it had to work at 32px (favicon), read clearly at 36px (navbar), and look sharp at 180px (Apple touch icon). The SVG is built with proportional coordinates — every element is calculated as a ratio of the icon size — so it scales perfectly at any size without a separate file for each context.

The drawing order matters in SVG: the stem and point are drawn first, then the circle on top. This lets the circle cleanly overlap the stem join without any gap or background bleed — a small detail that makes the difference between a logo that looks built and one that looks designed.

The design works at every size:
- Full wordmark in the navbar
- Contained icon as the favicon
- Apple touch icon for mobile bookmarks
- Embedded in the social sharing OG image

## The Design System

Rather than using a component library, I built a lightweight design system using CSS custom properties. Every color, font, and spacing decision lives in `:root` variables, which means changing the brand color site-wide is a one-line edit.

The grid background on hero sections — subtle CSS lines at `rgba(123,175,212,0.06)` — adds depth without competing with content. It fades out using a CSS mask gradient so it blends naturally into the page below.

The glow effect at the top of each hero section uses a `radial-gradient` on an absolutely positioned div. It's Duke Blue bleeding in from above — atmospheric without being distracting.

## The Homepage: StoryBrand Framework

The original homepage was functional but flat — a hero, a services snapshot, and a trust bar. Good structure, no narrative.

I rebuilt it using Donald Miller's **StoryBrand framework**, which flips the typical website mistake: instead of leading with "here's who we are," you lead with the customer's problem and position yourself as the guide who helps them win.

The full eight-section structure:

1. **Hero** — The customer's problem, stated boldly. *"Your next customer just searched Google. They found your competitor."*
2. **Problem** — Three pain points NWA contractors actually feel: invisible on Google Maps, outdated website, relying on word of mouth
3. **Guide** — Chad's story and credibility. Photo, timeline, stats — human before professional
4. **Plan** — Three steps: Talk, Plan & Build, Optimize & Grow
5. **What I Offer** — Six services with pill tags, honest about what's a core strength vs. growing skill
6. **Success vs. Failure** — Two columns showing what life looks like with and without taking action
7. **Transparency Pledge** — Four specific commitments, signed by the founder
8. **Final CTA** — An inline form, not just a button. Lower friction, higher conversion intent

Every section uses scroll-triggered animations via the browser's `IntersectionObserver` API — no external animation library needed. Each section fades up as it enters the viewport, with staggered delays on card grids. Think of it like a motion sensor that fires a CSS transition instead of a light.

## The Blog System

The blog runs on **MDX** — Markdown files with the ability to embed React components. Each post lives as a `.mdx` file in a `src/posts` folder. Next.js reads those files at build time and generates pages automatically.

This means adding a blog post is as simple as creating a new file and writing in Markdown. No database, no CMS login, no plugins to update. Version-controlled blog posts are a feature, not a limitation.

Each post page includes:
- A feature image if provided in the front matter
- Tag pills pulled from the post metadata
- An author avatar and byline pulled from the headshot
- A full author bio card at the bottom linking to the About page
- Post navigation — back to blog and a "Work With Chad" CTA

**Time taken: about 2 hours** to build the full blog system including the index, individual post template, and MDX configuration.

## The SEO Foundation

This is where I spent the most deliberate time. I refused to launch without proper SEO foundations — it would be embarrassing to build websites for contractors without practicing what I preach.

### Metadata and Open Graph

Every page has a unique title tag and meta description. The root layout uses Next.js's metadata system with a template that automatically appends the brand name. Open Graph tags ensure the site looks great when shared — with a dynamic branded preview image generated by Next.js's `ImageResponse` API at build time.

The OG image includes the grid background, Duke Blue glow, accent bars, the hero headline, and the logo. It's generated as a PNG at exactly 1200×630px — the standard social sharing size.

### LocalBusiness schema markup

`ProfessionalService` schema injected as JSON-LD in the document head. Tells Google the business name, URL, phone, email, address, service areas, and service types — all in a structured format Google can parse directly without inferring it from page content.

### Sitemap and robots.txt

Both auto-generated by Next.js from simple JavaScript files. The sitemap updates itself every time a new blog post or portfolio entry is published. No manual updates needed, ever.

### Dynamic favicon

Built with Next.js's `ImageResponse` — the same API used for OG images. The favicon, Apple touch icon, and OG image are all generated from the same design system, ensuring they're always consistent.

**Time taken: about 2 hours** for the full SEO setup.

## Mobile Responsiveness

The site is fully responsive with a custom hamburger menu that slides in a drawer from the right. Key technical details:

- The navbar becomes sticky on scroll and gains a frosted glass effect using `backdrop-filter: blur`
- The hamburger icon animates its three bars into an X when the drawer is open
- The body scroll is locked when the drawer is open using `document.body.style.overflow`
- Each drawer link animates in with a staggered delay using CSS `transitionDelay`
- Two-column grid sections collapse to single column at 768px using CSS media queries

Rather than a separate mobile stylesheet, I used utility class names (`two-col`, `three-col`, `form-row`) as hooks for media queries. Think of it like labels on boxes — the media query handles the stacking globally for any element with that class.

## Lessons Learned

### 1. Next.js 15 changed how params work

When building the blog post template, I hit an error I didn't expect — in Next.js 15, route params are now a Promise that needs to be awaited. In previous versions you could access `params.slug` directly. Now you need `const { slug } = await params`. A small change but it catches you off guard if you're following older tutorials.

### 2. Build the SEO foundation before launch

It's tempting to deploy and add SEO later. Don't. Schema markup, sitemaps, and metadata are structural — retrofitting them is harder than building them in from the start. Treat SEO like the foundation of a house, not the paint job.

### 3. Split server and client components deliberately

Next.js 15's App Router is strict about the boundary between server components (which can read files and fetch data) and client components (which can use React hooks and browser APIs). Several pages needed to be split into a server component that fetches data and a client component that handles animations. Think of it like a kitchen and a dining room — the kitchen preps the food, the dining room presents it.

### 4. `ImageResponse` requires `display: flex` on every element

The API that generates OG images and favicons is more restrictive than regular CSS — every `<div>` must have `display: "flex"` explicitly set, even decorative ones. Missing it causes a 500 error with no obvious message. Learned this the hard way.

### 5. Font pairing makes more difference than you think

I went through three font combinations — Poppins, then Cabinet Grotesk + Satoshi, and finally Fraunces + Urbanist. The last pairing transformed the feel of the site. Fraunces's editorial character and subtle italic style add a personality that purely geometric fonts can't achieve. Don't settle on fonts early — they're worth iterating on.

### 6. StoryBrand is a framework, not a template

The StoryBrand structure is powerful but needs to be adapted for your specific audience and positioning. A contractor-focused startup that leads with radical transparency needs a different tone than a polished agency. The framework gave us the skeleton — the voice and personality had to come from the brand guidelines.
