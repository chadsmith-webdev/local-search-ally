# How I Built the Local Search Ally Website: A Full Case Study

I want to be upfront about something before we get into this: this case study is about my own website. Not a client's. Mine.

That might seem like a strange way to start a portfolio piece. But it's actually the most honest thing I can do — because my brand is built on proof over claims, and I had to prove this to myself before I could offer it to anyone else.

Here's the full story.

---

## The Previous Version (And Why I Scrapped It)

The first version of localsearchally.com was built on Webflow. I built it myself, and at the time I didn't really know what I was doing. It showed.

The problems weren't catastrophic, but they were real:

- **Poor mobile experience** — the layout didn't translate well to smaller screens
- **Missing key pages and content** — no clear services structure, no about page that told the real story, no blog to build topical authority

I took it down. Not because I was embarrassed, but because I knew it was doing more harm than good. An SEO consultant with a bad website is like a plumber with a leaky house. The credibility math doesn't work.

So I went back to school.

---

## The Decision to Rebuild from Scratch

I spent time learning proper web development before touching the rebuild. The goal wasn't just a better-looking site — it was a site that would actually function as a business asset. That meant:

1. Fast load times (Core Web Vitals matter for rankings)
2. Built-in SEO architecture from the ground up
3. A blog that could compound over time
4. No dependency on a drag-and-drop platform I didn't fully control

That last point matters more than people realize. Page builders and website platforms abstract away control. When you're selling SEO, you need to own your own technical foundation.

---

## The Tech Stack

### Next.js 15

I chose Next.js 15 over other frameworks for a few specific reasons:

- **Server-side rendering** — pages are rendered on the server, which means faster initial load and better crawlability for Google
- **Static generation** — blog posts and portfolio pages can be pre-built at deploy time, which is as fast as it gets
- **Image optimization** — Next.js handles image compression and lazy loading automatically
- **Strong SEO primitives** — metadata API, canonical URLs, sitemap generation are all built in

WordPress was the obvious alternative. I ruled it out deliberately. WordPress sites are slower by default, require constant plugin maintenance, and the performance overhead is real. For a site where performance *is* the product demonstration, that didn't make sense.

### Vercel

Deploying on Vercel was a straightforward call. It's built by the same team that built Next.js, which means zero configuration friction. Edge network delivery, automatic HTTPS, preview deployments on every push. It's the right tool for this stack.

### MDX for Blog Content

Blog posts are written in MDX — Markdown with the ability to embed React components where needed. This means content lives as plain files in the codebase, not in a database. Simple, version-controlled, portable.

### Built-in SEO Architecture

Every page has:

- Unique, keyword-targeted `<title>` and `<meta description>` tags
- Proper heading hierarchy (one H1, logical H2/H3 structure)
- Canonical URLs to prevent duplicate content
- Schema markup for local business structured data
- An auto-generated XML sitemap
- Open Graph tags for social sharing

None of this was bolted on after the fact. It was designed in from day one.

## Performance Results

Here's where I'll be straight with you.

**Desktop PageSpeed score: 90–100** ✓  
**Mobile PageSpeed score: 70–79** — room to improve

The desktop score is where I want it. The mobile score is honest: 70–79 is passing but not excellent. The culprit is font loading — loading two variable fonts (Fraunces and Urbanist) from Google Fonts adds render-blocking time on mobile connections.

The fix is straightforward: self-host the fonts or switch to `font-display: optional`. I'll update this case study when that's done. That's what transparency looks like in practice — not hiding the gap, and committing to close it.

## The SEO Strategy

Building a technically clean site is necessary but not sufficient. You also need a content and keyword strategy.

For this site, the approach is straightforward:

**Target audience:** NWA contractor business owners searching for help with their online presence  
**Primary keyword cluster:** "local SEO for contractors NWA," "contractor SEO Northwest Arkansas," "[trade] SEO Bentonville/Fayetteville/Rogers"  
**Content strategy:** Blog posts that document real work, real results, and real process — not generic SEO advice

The blog exists for two reasons. First, topical authority — consistent content signals to Google that this site knows its subject. Second, trust — contractors who read a post about how I built this site understand my process before they ever talk to me.

Google Search Console is set up and indexing. It's too early for meaningful ranking data, but the foundation is right. I'll document results publicly as they come in.

---

## What I'd Do Differently

**Mobile performance from day one.** I'd self-host fonts before launch, not after. The 10–15 point gap between desktop and mobile scores is a known issue with Google Fonts loading, and it's fixable.

**More content at launch.** The site launched with one blog post and three placeholder portfolio items. Google rewards sites that launch with depth. Next time, I'd write five posts before going live.
